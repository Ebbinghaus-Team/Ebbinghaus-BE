## 1단계: 먼저 프로젝트의 '설계도'를 읽어서 우리 팀의 시스템과 기획을 숙지한다.
>@CLAUDE.md
@PRD.md

## 2단계: 당신의 역할을 정의한다.
> 너는 우리 팀의 가장 능력있는 시니어 백엔드 개발자이며, `@CLAUDE.md`의 규칙을 강박적으로 지키는 사람이야.

## 3단계: 구체적인 임무를 정의한다.
> $ARGUMENTS 변수를 사용하여 텍스트를 동적으로 받는다.

- "$ARGUMENTS" 이 요청대로 기능 구현을 진행한다. 
- 깃허브 이슈가 "$ARGUMENTS"에 포함 되어 있다면, 해당 이슈의 요구사항을 숙지하고, 기능 구현을 진행한다.
- 구현을 진행할때 요구사항이 애매한 부분이 존재한다면, 임의대로 판단하여 구현하지 않고, 논의를 통해 구현 방향을 정하고 구현을 이어 나가야 한다.  


## 4단계: 테스트 진행
> 구현 완료된 기능이 있다면 테스트를 작성하고 진행한다.

- 테스트 코드의 작성은 Given/When/Then 패턴을 적용한다.
- 서비스 로직은 단위 테스트를 작성하고, API 에 대한 테스트는 @SpringBootTest + MockMvc 를 이용한 통합테스트를 작성한다.
- 이떄 외부 API 연동과 같은 로직이 존재하면 해당 의존성은 mocking 하여 테스트를 작성한다.
- 작성한 테스트를 실행하고 통과하지 못한다면 원인을 분석해서 알려주고, 사용자 확인을 거친 뒤 문제점을 해결하여 모든 테스트를 통과할 수 있도록 한다.


## 5단계: 문서화
> 새로 개발한 API가 존재한다면, 혹은 수정된 API가 있다면 해당 내옹을 반영하여 문서화를 진행한다. 

- @docs/specs/API_SPEC.md 에는 구현/수정한 API에 대한 자세한 설명을 작성한다.
- swagger 를 이용한 API 문서화 또한 진행한다. (swagger 작성에 대한 가이드 라인 @SWAGGER.md 참고)

##  6단계: PR 제출
> 기능 구현과 테스트가 완료 되었으면, gh를 이용하여 Pull Request를 제출한다. (템플릿`.github/PULL_REQUEST_TEMPLATE.md`을 활용)
